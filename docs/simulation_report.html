🎨 Processus Design Thinking — 7 Étapes
Méthodologie appliquée pour concevoir HealthGuard IA centré sur l'utilisateur en contexte contraint
1

Empathize — Immersion terrain

Interviews avec 8 agents de santé communautaires (ASC) dans les districts de Ngaoundéré Rural (Adamaoua) et Bertoua (Est). Observation directe sur 3 semaines.

  • Connectivité réseau disponible < 30% du temps dans les zones cibles
  • Taux d'alphabétisation numérique : 65% des ASC sans expérience smartphone
  • Patientèle principale : enfants < 5 ans (40%), femmes enceintes (25%)
  • Maladies traitées : paludisme (45%), IRA (22%), malnutrition (13%), diarrhée (12%), TB (8%)
  • Douleur principale : "Je ne suis pas sûr(e) du diagnostic quand les signes sont mixtes"
2

Define — Définition du problème

HMW (How Might We) retenu : "Comment pourrions-nous aider un ASC peu expérimenté à prendre des décisions cliniques sûres pour des patients vulnérables, sans connexion internet, avec un téléphone Android d'entrée de gamme ?"

  • Contrainte critique #1 : Offline-first absolu — aucune décision ne doit dépendre du réseau
  • Contrainte critique #2 : Faux négatifs à coût élevé — paludisme grave non détecté = décès potentiel
  • Contrainte critique #3 : Interface simple — max 3 taps pour démarrer une consultation
  • Contrainte réglementaire : Conformité Loi N°96/03 Cameroun — secret médical, données personnelles
3

Ideate — Génération d'idées

Session de co-création (3h) avec 4 ASC, 2 médecins superviseurs, 1 épidémiologiste district.

  • Idée retenue : Arbre décisionnel interactif + validation ML — "L'arbre pose les bonnes questions, le ML confirme"
  • Rejetée : Chatbot vocal (coût infrastructure, latence)
  • Rejetée : Formulaire libre (trop d'erreurs de saisie sans guidage)
  • Innovation : Code couleur ROUGE/ORANGE/VERT — immédiatement compréhensible sans lecture
  • Innovation : PIN 6 chiffres + accès urgence anonyme pour les situations critiques
4

Prototype — Construction du prototype

7 écrans HTML Mobile First (720×1560px, simulateur Android) créés en 3 itérations.

  • E1 — Connexion : Clavier PIN 6 chiffres, accès urgence, biométrie désactivable
  • E2 — Tableau de bord : Stats du jour, actions rapides, alerte sync > 48h
  • E3 — Consultation : Formulaire sympto + constantes vitales avec alertes visuelles
  • E4 — Arbre décisionnel : Questions guidées avec pré-remplissage automatique
  • E5 — Résultat : Diagnostic + résumé 3 points + traitement + contre-indications
  • E6 — Dossier patient : Recherche, historique, suivi longitudinal
  • E7 — Paramètres : Sync, sécurité PIN, export données, version modèle
5

Test — Validation utilisateurs

Tests avec 4 ASC sur scénarios cliniques réels simulés. Méthode : think-aloud + observation.

  • Résultat : 95% des diagnostics corrects vs protocole PCIME superviseur
  • Friction identifiée : Saisie PB en mm peu intuitive → ajout jauge visuelle colorée
  • Friction identifiée : "Diagnostic différentiel" incompris → renommé "À ne pas exclure"
  • Satisfaction : Note moyenne 4.4/5 sur la clarté des recommandations
  • Temps moyen consultation : 3 min 42s (cible : < 5 min) ✓
6

Implement — Développement technique

12 modules implémentés en Python (backend) + HTML/CSS/JS (prototype) sur 4 semaines.

  • Module 0 : Structure projet + schéma SQLite + chiffrement AES-256
  • Module 1 : 5 arbres décisionnels JSON (format FHIR-like)
  • Module 2 : tree_navigator.py + agrégateur hybride arbre/ML
  • Module 3 : XGBoost : 5000 cas synthétiques, 24 features, quantification INT8
  • Module 4 : FastAPI — 7 endpoints REST + Pydantic validation
  • Module 5 : Sécurité : Argon2id PIN, audit trail SHA-256, verrouillage exponentiel
  • Module 6 : Prototype HTML — 7 écrans Material Design 3
  • Modules 7-9 : Tests unitaires (134 tests, 85% couverture)
7

Scale — Déploiement et passage à l'échelle

Feuille de route pour le déploiement pilote dans 5 CSI du district de Ngaoundéré (Q2 2025).

  • Phase pilote (3 mois) : 20 ASC, 5 CSI, avec supervision médecin district
  • Métriques de succès : Taux de détection paludisme grave ≥ 90%, satisfaction ASC ≥ 4/5
  • Infrastructure sync : Serveur district (Ubuntu 22.04 + PostgreSQL + FastAPI)
  • Mise à jour modèle : Ré-entraînement trimestriel sur données collectées
  • Extension prévue : 50 CSI supplémentaires (Adamaoua + Est) en Q4 2025
  • Certification : Validation protocoles par l'OMS bureau Yaoundé en cours

🔐 Architecture de Sécurité — Modèle STRIDE

S — Spoofing (Usurpation)

Argon2id PIN (time=3, mem=64MB, par=4). Timeout session 5 min. Verrouillage exponentiel après 5 échecs (délai 5s → 25s → 125s → 625s).

T — Tampering (Falsification)

AES-256-CBC + Encrypt-then-MAC (HMAC-SHA256). Vérification intégrité avant tout déchiffrement. Schéma BD avec contraintes d'intégrité.

R — Repudiation (Répudiation)

Audit trail avec chaîne de hachage SHA-256 (lightweight blockchain). Chaque log contient le hash du log précédent — toute modification détectable.

I — Information Disclosure

Toutes les données sensibles (noms, symptômes, diagnostics) chiffrées AES-256 au repos. Clé dérivée du PIN via PBKDF2-SHA256 (256 000 itérations).

D — Denial of Service

Verrouillage après 5 tentatives PIN. SQLite WAL pour performance sous charge. Sync queue avec retry max 3 pour éviter boucles infinies.

E — Elevation of Privilege

Accès urgence limité aux consultations (non associées à un agent). Pas de rôle administrateur sur mobile. API localhost uniquement.

🏁 Conclusion et Impact Attendu

HealthGuard IA répond aux contraintes les plus sévères du terrain : zéro dépendance réseau pour le diagnostic, données sensibles protégées par chiffrement de niveau hospitalier, et interface conçue pour des utilisateurs à faible littératie numérique.
