# Audit HealthGuard IA



## Manquements critiques

- L'exigence `SQLite chiffré (SQLCipher)` n'est pas implémentée. Le code ouvre un SQLite standard sans `PRAGMA key`, sans binding SQLCipher, et les dépendances ne contiennent aucun package SQLCipher ([src/database/schema.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/database/schema.py:33), [requirements.txt](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/requirements.txt:23)).
- Le chiffrement au repos est incomplet malgré la documentation. `date_naissance`, `sexe`, `village_code`, `agent_id`, `date_heure` et les métadonnées restent stockés en clair. La colonne `date_naissance` est même décrite comme chiffrée dans le schéma mais insérée en clair par l'API ([data/db/schema.sql](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/data/db/schema.sql:9), [src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:221), [src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:226)).
- La clé de chiffrement locale dépend d'un PIN global statique avec valeur par défaut triviale `123456` et d'un sel stable connu. Cela annule en pratique l'objectif de protection forte au repos ([src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:134), [.env.example](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/.env.example:10)).
- Le système accepte un mode `urgence_anonyme` sans identification réelle, incompatible avec un audit trail médical opposable ([prototype/screens/e1_login.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e1_login.html:359)).
- L'audit trail n'est pas exécuté sur les actions métier. Aucun endpoint patient, diagnostic, sync ou login n'appelle `log_action()` malgré l'existence du module ([src/database/audit.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/database/audit.py:33), [src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:427)).
- Le transport annoncé `TLS 1.3` n'est pas garanti. Le code applicatif synchronise en `psycopg2.connect()` sans configuration TLS explicite, et la configuration Nginx autorise encore TLS 1.2 ([src/database/postgres_sync.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/database/postgres_sync.py:21), [healthguard_nginx.conf](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/healthguard_nginx.conf:14)).

## Failles de sécurité

- Les hashes PIN des agents sont stockés dans `localStorage`, donc récupérables par tout JavaScript du contexte navigateur ou par une compromission du terminal ([prototype/screens/e1_login.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e1_login.html:186), [prototype/screens/e1_login.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e1_login.html:241)).
- Les PIN de démonstration sont affichés à l'écran et conservés côté client (`246810`, `135791`) ([prototype/screens/e1_login.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e1_login.html:171), [prototype/screens/e1_login.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e1_login.html:249)).
- Le module TLS est une simulation avec secret HMAC embarqué en dur; il ne protège aucun flux réel ([src/security/tls_sync.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/security/tls_sync.py:14), [src/security/tls_sync.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/security/tls_sync.py:42)).
- La base d'export PostgreSQL versionnée contient des données cliniques en clair, y compris symptômes et recommandations ([data/db/healthguard_full_export.sql](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/data/db/healthguard_full_export.sql:253), [data/db/healthguard_full_export.sql](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/data/db/healthguard_full_export.sql:270)).
- Les dates de naissance sont persistées en clair dans la base active. Inspection locale: `patients_sample` contient par exemple `1990-05-15`.
- Le service worker ne met pas en cache l'API mais laisse le frontend appeler le réseau dès qu'il est disponible; aucune stratégie de protection des données API n'est présente ([prototype/sw.js](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/sw.js:42)).

## Problèmes d’architecture

- Le design attendu parlait d'un moteur hybride `PCIME + XGBoost` orchestré côté backend. En pratique, le frontend embarque un second moteur heuristique `runDiagnosticLocal()` qui ne réutilise ni les arbres JSON ni l'agrégateur Python, créant deux vérités médicales concurrentes ([prototype/screens/e5_result.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e5_result.html:142), [src/decision_engine/aggregator.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/decision_engine/aggregator.py:37)).
- L'endpoint `/api/v1/diagnostic/tree-step` ne pilote aucun état de session clinique réel; il renvoie uniquement un message factice ([src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:497)).
- La synchronisation dépend directement de PostgreSQL, pas d'un canal sécurisé opportuniste clairement encapsulé. Le composant `tls_sync.py` n'est pas utilisé par la chaîne réelle ([src/database/postgres_sync.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/database/postgres_sync.py:31), [src/security/tls_sync.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/security/tls_sync.py:63)).
- Le schéma déclare une table `traitements`, mais les traitements ne sont jamais persistés comme entités métier séparées. Ils restent inclus dans du JSON de recommandation ([data/db/schema.sql](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/data/db/schema.sql:41), [src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:266)).
- Le module `postgres_sync.py` contient du code mort ou incohérent. `patients_pending` est calculé puis jamais utilisé ([src/database/postgres_sync.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/database/postgres_sync.py:57)).
- Le backend déchiffre le nom patient pour l'afficher dans plusieurs endpoints sans séparation claire entre couches de stockage, domaine et exposition API ([src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:160), [src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:553)).

## Non-respect des contraintes offline-first

- Le parcours offline n'est pas unifié. Hors ligne, l'application bascule sur `localStorage` et sur un moteur JS de secours; en ligne, elle bascule sur l'API locale. Les diagnostics peuvent donc diverger selon l'état réseau ([prototype/screens/e5_result.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e5_result.html:473)).
- Le service worker ne rend pas l'API disponible hors ligne; il ignore explicitement les requêtes `/api/v1/` ([prototype/sw.js](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/sw.js:43)).
- Le tableau de bord, les patients et la synchronisation tentent d'appeler le réseau navigateur avec `fetch()` et utilisent des fallbacks partiels, pas un socle local unique ([prototype/screens/e2_dashboard.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e2_dashboard.html:192), [prototype/screens/e6_patient_record.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e6_patient_record.html:174)).
- La page de paramètres exporte et purge directement `localStorage`, preuve que ce stockage reste une source primaire de données de consultation ([prototype/screens/e7_settings.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e7_settings.html:477), [prototype/screens/e7_settings.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e7_settings.html:498)).
- La synchronisation “opportuniste sécurisée” annoncée n'est pas autonome. Elle repose sur un déclenchement manuel `Synchroniser maintenant` ([prototype/screens/e7_settings.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e7_settings.html:294)).

## Problèmes liés au modèle IA

- Le modèle réellement livré est un `xgboost.sklearn.XGBClassifier` sérialisé en pickle, pas un modèle TinyML embarqué prêt smartphone. Inspection runtime: type `xgboost.sklearn.XGBClassifier`, taille fichier `1 559 218` octets.
- La contrainte de taille `< 8 Mo` semble satisfaite sur le fichier pickle actuel, mais la contrainte de format embarqué n'est pas satisfaite: aucun artefact `model_healthguard.tflite` n'est livré dans `data/ml/`.
- La conversion TFLite est elle-même non fiable: si TensorFlow est absent, le code “simule” la conversion et retourne artificiellement un succès avec `50 ms` d'inférence ([src/ml/convert_tflite.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/ml/convert_tflite.py:31), [src/ml/convert_tflite.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/ml/convert_tflite.py:170)).
- Les dépendances TFLite/TensorFlow sont commentées comme optionnelles, ce qui confirme que le chemin de déploiement embarqué n'est pas industrialisé ([requirements.txt](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/requirements.txt:37)).
- Les performances cliniques du modèle reposent sur un dataset synthétique généré localement, pas sur des données cliniques réelles validées terrain ([src/ml/data_generator.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/ml/data_generator.py:1), [src/ml/train_model.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/ml/train_model.py:88)).
- Le chargement de modèle peut déclencher un entraînement automatique en production si le pickle est absent, ce qui est incompatible avec une application embarquée déterministe et à faible ressource ([src/ml/inference.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/ml/inference.py:48)).
- L'imputation des données manquantes remplace des mesures critiques par des valeurs normales par défaut, en particulier `spo2_percent=97.0`, ce qui peut masquer une détresse respiratoire quand l'oxymètre est absent ([src/ml/inference.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/ml/inference.py:83)).
- Plusieurs branches de l'arbre utilisent des suppositions cliniques codées en dur (`N2=True`, `N3=True` pour tuberculose), ce qui dégrade la traçabilité clinique du diagnostic ([src/decision_engine/tree_navigator.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/decision_engine/tree_navigator.py:423)).
- L'agrégateur calcule une “probabilité combinée” par formules heuristiques dépendant de la gravité de l'arbre et non d'une calibration probabiliste validée ([src/decision_engine/aggregator.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/decision_engine/aggregator.py:79)).

## Risques sur les données médicales

- `localStorage` stocke les consultations, noms patients, symptômes, diagnostics, suivi et état de synchronisation en clair côté navigateur ([prototype/screens/e5_result.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e5_result.html:375)).
- L'export JSON depuis les paramètres permet l'exfiltration complète des consultations navigateur sans contrôle d'accès additionnel ([prototype/screens/e7_settings.html](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/prototype/screens/e7_settings.html:477)).
- Le mécanisme `sync/from-browser` importe dans la base locale des données construites côté navigateur, y compris du diagnostic déjà calculé hors backend. Cela ouvre la voie à l'injection de dossiers médicaux non fiables ([src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:610)).
- Le patient ID de secours dérivé par `uuid5` à partir de `nom-age-sexe` est déterministe. Cela crée un risque de ré-identification et de collision sémantique ([src/api/app.py](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/src/api/app.py:632)).
- Le schéma considère `village_code` comme pseudo-anonyme alors qu'en contexte rural il reste fortement identifiant lorsqu'il est corrélé à l'âge, au sexe et au diagnostic ([data/db/schema.sql](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/data/db/schema.sql:11)).
- `healthguard_full_export.sql` contient au moins des patients nommés `Moussa Aliou`, `Ibrahim Mahamat`, `Fatima Oumar` et des symptômes associés en clair ([data/db/healthguard_full_export.sql](/D:/ARCHIVES_GENERALES/01_PROJETS_ET_TRAVAIL/Project/healthguad/data/db/healthguard_full_export.sql:327)).
